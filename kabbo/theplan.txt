## Context

Kabbo is a publication pipeline tracker named after ||kabbo, the |Xam storyteller.
It currently lets individual researchers drag cards through seven stages
(Idea -> Published) and has a functional teams layer (invitations, visibility
settings, team workspaces, analytics, report export). An MCP server with 6
tools provides Claude Code integration.

The goal is to transform Kabbo from a free personal tool into the one-stop
platform for the entire research publication workflow — from idea generation
to paper dissemination — with a clear monetisation path:

- Free tier: Individual researchers. Full pipeline, MCP integration, Claude
  Code skill.
- Paid tier (Team Leader): Create and manage research teams, see team
  analytics, set targets, track bottlenecks, export reports. Team members
  stay on free — only the leader pays.
- Future scaling: Team -> Department -> Faculty -> School -> University.

The strategic differentiator is twofold: (1) Claude Code / Codex integration
via MCP — no other academic tool offers this, and (2) the only tool that gives
research leaders visibility into the pre-publication pipeline, where papers are
invisible to bibliometric systems.

## Pre-wave: Infrastructure (before Wave 1 ships to users)

### Resend — production email

Kabbo currently has email confirmation disabled because there's no production
email provider. Supabase's built-in mailer is rate-limited and uses a generic
sender address. Setting up Resend fixes this.

Steps:
1. Create a Resend account at resend.com (free tier: 3,000 emails/month)
2. Add and verify kabbo.app domain in Resend (add DNS records in Porkbun):
   - MX record for receiving bounces
   - TXT record for SPF
   - CNAME records for DKIM (Resend provides these)
3. Generate a Resend API key
4. Configure Supabase SMTP (dashboard > Auth > SMTP Settings):
   - Host: smtp.resend.com
   - Port: 465
   - Username: resend
   - Password: re_YOUR_RESEND_API_KEY
   - Sender name: Kabbo
   - Sender email: noreply@kabbo.app
5. Re-enable email confirmation in Supabase Auth settings:
   - Auth > Providers > Email > Enable email confirmations
6. Test: sign up with a fresh email, confirm the confirmation email arrives
   from noreply@kabbo.app with valid DKIM signature

Status: NOT DONE. User has a Resend account pending setup.

 #  | Tool                  | What it does                                               | Example prompt
----|-----------------------|------------------------------------------------------------|------------------------------------------

15  | add_note              | Append to publication notes without overwriting             | "Note on climate paper: reviewer 2 wants robustness checks"

Codex uses the same MCP protocol as Claude Code. The existing MCP server works

#### 1E. Smart Insights panel (non-Claude-Code users)

A collapsible InsightsPanel on the main dashboard with rule-based suggestions:

 Category         | Example                                              | Logic
------------------|------------------------------------------------------|-------------------------------
 Stalled          | "Colonial Wages has been in R&R for 92 days"         | now - last_stage_change > 30d
 Missing data     | "3 papers have no target year"                       | Check completionYear
 Momentum         | "You submitted 5 papers this year, up from 3 last"   | Stage transitions by year
 Quick wins       | "2 accepted papers — ready to mark Published?"       | Count accepted
 Celebration      | "You published 3 papers this year!"                  | Count published

This gives non-technical users a taste of what Claude Code offers, creating
pull toward the full integration.

### WAVE 3: THE PAID TIER — TEAM LEADER (2-3 weeks)

This is where revenue begins. The pricing model:

FREE (Individual Researcher)
- Full pipeline (all 7 stages)
- MCP / Claude Code / Codex integration
- Reminders, data sources, related papers
- CSV, ORCID, BibTeX import
- Join unlimited teams as a member
- Basic personal analytics

TEAM LEADER ($9/month or $89/year per team)
- Create and manage up to 1 research team (additional teams: same price each)
- Invite unlimited members (members stay free)
- Team dashboard with aggregate analytics
- Per-member pipeline visibility (what stages you can see)
- Publication targets (set goals, track progress against them)
- Bottleneck analysis (where do papers get stuck?)
- Team activity log
- Annual review export (per-member PDF/Excel)
- Grant progress view (papers grouped by grant)
- Priority support

Why $9/month: Low enough that a PI can expense it without procurement. High
enough to signal value. Annual discount encourages commitment. A department
head with 5 teams pays $45/month — still trivial against a research budget.

#### 3A. Stripe integration

- Create a Stripe account with two products: "Team Leader Monthly" ($9) and
  "Team Leader Annual" ($89)
- Use Stripe Checkout (hosted payment page — no PCI compliance burden)
- Stripe Customer Portal for subscription management (cancel, update card)
- Webhook: stripe-webhook edge function to handle checkout.session.completed,
  customer.subscription.updated, customer.subscription.deleted

#### 3B. Database changes

New table: subscriptions
  id uuid primary key
  user_id uuid references profiles(id)
  stripe_customer_id text
  stripe_subscription_id text
  status text ('active', 'canceled', 'past_due', 'trialing')
  plan text ('team_leader_monthly', 'team_leader_annual')
  current_period_end timestamptz
  created_at, updated_at timestamptz

New column on teams: subscription_id uuid references subscriptions(id)

#### 3C. Enforcement logic

- Creating a team checks subscriptions for an active subscription
- Existing teams (created before billing launch) get a 30-day grace period
- Team members are never billed — only the team creator
- If subscription lapses: team dashboard becomes read-only (data preserved,
  not deleted), team leader sees "Resubscribe to manage your team"

#### 3D. Team Leader features (the value proposition)

PUBLICATION TARGETS
- Team leader sets annual targets per member: "I expect 2 published papers and
  3 submissions this year"
- Dashboard shows progress-to-target with a simple bar chart
- New table: publication_targets (team_id, member_user_id, year,
  target_published, target_submitted, target_drafts)

BOTTLENECK ANALYSIS
- Visual heatmap: which stage holds the most papers, for how long?
- Per-member breakdown: "Dr X has 4 papers stuck in R&R averaging 120 days"
- Aggregate: "60% of your team's papers stall at the submission stage"
- Data source: stage_history JSON array already stored on every publication

STRAGGLER / SUPERSTAR DETECTION
- Rank team members by publication velocity (papers moving per quarter)
- Flag members with zero stage transitions in 60+ days
- Highlight members with unusually high throughput
- Present as a neutral "Activity Summary" — avoid punitive framing

TEAM ACTIVITY FEED
- Real-time feed of stage transitions across the team: "Alice moved 'Climate
  Paper' to Submitted — 2 hours ago"
- Uses existing activity_log table, filtered by team membership
- Accessible from team dashboard

ANNUAL REVIEW EXPORT (enhanced)
- Per-member PDF with: publications by stage, targets vs actuals,
  time-in-stage averages, activity timeline
- "Generate Annual Review for [Member] — [Year]" button in team dashboard
- Uses existing TeamReportExport.tsx pattern, extended with target data

GRANT PROGRESS VIEW
- Group papers by grant tag
- Show: total papers per grant, papers by stage, stalled papers, published
  output
- Useful for grant reporting: "NRF-2026 has produced 3 publications and has
  5 in pipeline"

### WAVE 4: DISSEMINATION TRACKING (2-4 weeks, after Wave 3)

Extend Kabbo beyond the pipeline exit (Published) to track what happens after
publication. This closes the loop from idea to impact.

#### 4A. Citation tracking

- Integrate OpenAlex API (free, no rate limit) or Semantic Scholar API
- For each published paper with a DOI: fetch citation count periodically
  (weekly cron job via Supabase pg_cron or a scheduled edge function)
- Display on PublicationCard: "12 citations" badge
- Team dashboard: citation velocity, h-index approximation, most-cited papers

#### 4B. Altmetric / media coverage

- Integrate Altmetric API (free tier available) or PlumX
- Track mentions in news, blogs, social media, policy documents
- Display Altmetric "donut" or a simplified score on published papers
- Team dashboard: "Papers in the media this month"
- Alternative (simpler, no API): let users manually log media appearances
  (title, outlet, URL, date) via a media_coverage table

#### 4C. Social media package generation

- Reuse the existing /olwsocial skill pattern
- When a paper is published: generate platform-specific social posts
  (LinkedIn, X/Bluesky, institutional newsletter)
- Store templates per team (some teams want formal, others casual)
- This is a natural extension of the ||kabbo "story" metaphor — every
  published paper deserves its own telling

### WAVE 5: SCALE FROM TEAM TO INSTITUTION (long-term, 3-6 months)

The hierarchy: Researcher -> Team -> Department -> Faculty -> University

#### 5A. Organisational hierarchy

New tables:

  organisation_teams (organisation_id, team_id)
  organisation_admins (organisation_id, user_id, role: 'admin'|'viewer')

A department head sees all teams in their department. A dean sees all
departments in their faculty. Roll-up analytics at each level.

#### 5B. Institutional pricing

 Tier          | Price       | What you get
---------------|-------------|----------------------------------------------
 Team Leader   | $9/month    | 1 team, unlimited members
 Department    | $49/month   | Up to 10 teams, department analytics, reviews
 Faculty       | $199/month  | Unlimited teams, faculty roll-up, branding
 University    | Custom      | Multi-faculty, SSO integration, support

#### 5C. SSO / institutional identity

- SAML or OpenID Connect integration for university single sign-on
- Auto-provisioning: university login -> placed in the right department
- This is the enterprise unlock — universities won't adopt without it

#### 5D. Performance evaluation framework (careful — defer unless requested)

- Team leader defines evaluation criteria (publication count, citations,
  grant success, mentoring load)
- System generates a draft evaluation report from Kabbo data
- Leader reviews and adjusts before sharing
- CRITICAL: never auto-share evaluations with evaluated members. The leader
  always controls what's shared.
- Position as "evidence-based review support" not "performance monitoring"

## Competitive positioning

What exists today for academics:
- Spreadsheets (Excel, Google Sheets) — no collaboration, no analytics, no AI
- Trello/Notion — generic, not built for research workflow
- Manuscript Central / ScholarOne — journal-side, not researcher-side
- ORCID — tracks published output only, not the pipeline
- Pure (Elsevier) — enterprise CRIS, expensive, heavyweight

Kabbo's unique position:
- The only tool that tracks the pre-publication pipeline (where papers are
  invisible to every other system)
- The only tool with Claude Code / Codex integration via MCP
- Free for individuals, affordable for team leaders
- Named after a storyteller who understood that every journey has stages,
  seasons, and waiting

The pitch (one sentence):
  Kabbo is where your research lives between the idea and the publication —
  and the only tool that lets you ask Claude how it's going.

shipped: it reads from `publications`, groups / counts / dates, and

### After 1E — pick from Wave 2 in order of impact

Wave 2A (Reminders) — highest user-value individual feature that reduces
churn the most. Unlocks the "Kabbo tells me what to do today" behaviour.

Wave 2D (CSV import) — single biggest onboarding friction reducer.
Academics track papers in Excel; getting them off Excel into Kabbo in one
drag is the killer import flow.

Wave 2E (ORCID import) — instant gratification for new users. Log in,
paste your ORCID, see your 40 published papers populate in 10 seconds.

Defer Wave 2B (data sources) and 2C (related papers) unless a user
specifically asks — the schema exists but the UX value is smaller.
